# The Main Problem You’re Running Into

You already identified it:

> “I feel like I’m losing the essence or important details later.”

This happens because:

- summaries are linear,
    
- thesis writing is comparative and relational.
    

You usually do _not_ need:

You usually _do_ need:

- “what this paper says about X compared to Y.”

That is not a summary problem.  
It is a **comparison matrix** problem.

Ah, jetzt wird’s spannend, denn genau hier entsteht das **Powerhouse von Obsidian**: die Netzwerk-Sicht und die Wiederverwendbarkeit. Ich erkläre Schritt für Schritt, wie du die Notes so verlinkst, dass Konzepte, Papers, Vergleiche und Research Questions miteinander “sprechen”.

---

## 1️⃣ Grundprinzip

- **Jede Note = eigenständige Einheit.**
    
- **Jede Concept Note = Knoten.**
    
- **Linke so viel wie nötig, wiederhole nicht alles in allen Notes.**
    
- Verlinke nur, wenn es **direkten Bezug** gibt, sonst entsteht nur Noise.
    

---

## 2️⃣ Arten von Links

1. **Paper → Concept**
    
    - In der Source Note eines Papers markierst du, welche Konzepte relevant sind.
        
    - Syntax: `[[Concept Note Name]]`
        
    
    Beispiel in „Brandt 2006 - Designing Exploratory Design Games.md“:
    
    ```markdown
    ## Relevant Concepts
    - [[Design Game Categorization]]
    - [[Boundary Objects]]
    - [[Tangible Materials]]
    ```
    
2. **Concept → Concept**
    
    - Wenn ein Konzept inhaltlich mit einem anderen verwandt ist, verlinke.
        
    - Beispiel:
        
    
    ```markdown
    # Boundary Objects
    Boundary objects unterstützen [[Co-Creation]] in Design Thinking Prozessen.
    ```
    
    Dann entsteht automatisch eine **bidirektionale Verbindung** in Obsidian.
    
3. **Concept → Comparison**
    
    - Wenn du Konzepte vergleichst, verlinke auf die Concept Notes.
        
    - Beispiel:
        
    
    ```markdown
    # DT Methods vs Design Games
    Im Ideate-Step unterstützen [[Scenario Games]] und [[Tangible Materials]] ähnliche Ziele.
    ```
    
4. **Comparison → Research Question**
    
    - Wenn eine Lücke sichtbar wird, verlinke auf die Research Question Note.
        
    - Beispiel:
        
    
    ```markdown
    # DT Methods vs Design Games
    Wir sehen: Im Define-Step gibt es wenige DG-Methoden → [[Potential for DG in Define Phase]].
    ```
    
5. **Concept → Research Question**
    
    - Wenn ein bestimmtes Konzept direkt eine Frage aufwirft.
        
    - Beispiel:
        
    
    ```markdown
    # Scenario Games
    Kann die Methode auch außerhalb IT eingesetzt werden? → [[DG outside IT]]
    ```
    

---

## 3️⃣ Struktur-Tipp für Links

- **Verwende sprechende Namen**:  
    `[[Scenario Games]]` ist besser als `[[Note1]]`.
    
- **Ergänze Tags** sparsam, damit Obsidian nicht überladen wird.
    
- **Nutze “Backlinks” aktiv**:
    
    - Wenn du auf eine Concept Note klickst, siehst du alle Papers, Comparisons, Research Questions, die auf sie verweisen.
        
    - Das ist mega praktisch für Synthese und Schreibarbeit.
        

---

## 4️⃣ Visualisierung

- **Graph View**:
    
    - Jedes Paper = Punkt
        
    - Jede Concept Note = Punkt
        
    - Jede Research Question = Punkt
        
    - Jede Linie = Link
        
- Du siehst auf einen Blick:
    
    - Welche Konzepte gut abgedeckt sind
        
    - Wo Lücken bestehen
        
    - Welche Papers mehrere Konzepte verbinden
        

---

## 5️⃣ Mini-Workflow für Links beim Lesen

1. Du liest Paper → markierst relevante Konzepte → `[[Concept Note]]` einfügen.
    
2. Während Concept Notes wachsen → prüfe: gibt es verwandte Konzepte? → Link setzen.
    
3. Vergleichs-Tabelle bauen → Link zu Concept Notes + Research Questions.
    
4. Thesis schreiben → direkt Concept Notes, Comparison Notes und Research Questions referenzieren.
